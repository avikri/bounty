Bounty · Design spec v0.1 · May 2026

A social wager app for groups who can't take a dare.

A gamified, group-private bounty board: post a dare, set a price, watch your friends squirm. This document defines the screens, visual system, Angular component tree, and Firestore data model — followed by mid-fi mockups of every primary screen.

01

User flow & screen breakdown

Twelve screens cover the full lifecycle. Numbered roughly in the order a new user would meet them. OP = Original Poster (the user who created the bounty).

01
Login / Auth route: /login

Third-party SSO only (Google, Apple). Gates the entire app behind AuthGuard.

  • Elements: wordmark, one-line value prop, "Continue with Google", "Continue with Apple", T&C link.
  • Actions: sign in → redirect to previous deep link or /groups.
02
Group hub route: /groups

Landing screen post-login. Shows every group the user belongs to with live activity counts.

  • Elements: group cards (name, member count, unread bounty count, your rank), CTA "Create group", CTA "Join by code".
  • Actions: tap card → group, create new, join existing via 6-char invite code.
03
Active bounty feed route: /g/:groupId

The main screen. Real-time list of active bounties in the selected group, sorted newest first by default.

  • Elements: sticky group header (name, member chips, leaderboard link), filter pills (All · Available · Mine · Claimed), bounty cards, FAB "+ Post bounty".
  • Actions: filter, tap card → detail, FAB → create flow, pull-to-refresh.
04
Bounty detail route: /g/:groupId/b/:bountyId

Full detail of one bounty plus its current state machine context.

  • Elements: title, description, price chip, expiration countdown, state badge with icon, OP avatar + name, current claimant (if any), activity timeline, contextual CTA.
  • Actions vary by viewer + state: Claim · Submit proof · Withdraw claim · Approve · Reject · Delete (OP only, while Available).
05
Create bounty route: /g/:groupId/new

Single-step form. Keep friction low — most bounties are written on a whim.

  • Elements: Title, Description (multiline), Price ($), Expiration (date+time picker, sensible default = 7 days), proof-required toggle, "Post" CTA.
  • Actions: submit, save draft (auto), cancel.
06
Claim & proof submission route: /g/:groupId/b/:bountyId/submit

Where the claimant proves they actually did the thing. Required before the OP can review.

  • Elements: bounty summary, photo/video upload (up to 3), short note field, "Submit for review" CTA.
  • Actions: upload, write note, submit → bounty enters pending_review.
07
OP review queue route: /reviews

Cross-group inbox of every claim awaiting your approval (you posted the bounty). Notification badge in tab bar.

  • Elements: queue of pending reviews grouped by group, each row shows claimant, bounty title, price, proof preview.
  • Actions: tap row → review modal, approve, reject (with optional reason).
08
Review detail (approve/reject) modal over /reviews

Focused decision view. Shows full proof at viewable size.

  • Elements: proof gallery, claimant note, bounty rules recap, payout preview ("+30 pts & you owe @nora $30"), Reject / Approve buttons.
  • Actions: approve (success path), reject with reason (failure path).
09
Leaderboard route: /g/:groupId/leaderboard

Per-group standings. Real-time, denormalized for speed.

  • Elements: top-3 podium, ranked list (avatar, name, points, W/L, net IOU), time-range tabs (All time · This month · This week).
  • Actions: tap row → user profile, share leaderboard image.
10
User profile route: /u/:userId

Per-user record. Public within shared groups.

  • Elements: avatar, name, total points, win/loss/abandon counts, biggest score, history list, net IOU per counterparty.
  • Actions: view bounty history, "Mark as settled" on individual IOUs (both parties must confirm).
11
Group settings route: /g/:groupId/settings

Admin tools for the group owner; lighter view for members.

  • Elements: group name, avatar/emoji, invite code (regenerable), member list with roles, leave/delete group, default bounty expiration.
  • Actions: rename, regenerate invite, promote/remove member, leave, delete (owner only).
12
Notifications route: /inbox

Cross-group activity stream: claims on your bounties, decisions on your claims, leaderboard rank changes, expirations.

  • Elements: grouped list by day, unread state, deep-link to each item.
  • Actions: tap to deep-link, mark all read.
02

UX / UI recommendations

Calm chrome, loud moments. The layout stays clean and library-grade so people can scan a feed quickly; the gamified beats (price, points, podium, state changes) are where we turn up the volume.

Palette

Warm cream surfaces over a soft-ink near-black. Tangerine is the brand accent and primary CTA. State colors share a single chroma so badges sit comfortably next to one another.

Cream--bg
Ink--ink
Tangerine--primary
Win green--success
Fail red--danger
Review purple--purple

Typography

Bricolage Grotesque

Display + section headers + prices. A friendly grotesque with optical sizing — feels playful at large sizes, settles down at small ones.

Plus Jakarta Sans

Body, UI labels, buttons. Rounded terminals soften the interface without going full marshmallow. Weights 400/500/600/700/800.

Numerals use the display face — prices and points are loud on purpose. Mono (JetBrains Mono) reserved for invite codes, timestamps, and dev-flavored UI accents.

Layout structure

  • Mobile (primary): single-column feed, sticky group header, bottom tab bar (Feed · Reviews · Post · Leaderboard · Profile).
  • Desktop: three-pane (left nav · feed · right rail w/ leaderboard + activity). Same primitives, more density.
  • Cards over chrome: bounty cards are the unit of composition — feed, profile, reviews, and leaderboard all reuse them.
  • Spacing rhythm: 4 / 8 / 12 / 16 / 22 / 36 px scale. Radii: 10 / 14 / 20 / 28.

Distinguishing bounty states

Five visual states, each with a distinct icon + soft tint pair. The icon is the load-bearing signal (works in greyscale, accessible, scannable at thumb distance); the tint just reinforces it. Never use color alone.

Available Open for anyone to claim. Blue target.
In progress Claimed, working on it. Lightning bolt.
Pending review OP needs to decide. Watching eye.
Successful Approved, points awarded. Checkmark.
Failed Rejected or expired. Points docked.

Gamification accents

  • Points popups: celebratory toast on approval (+30 pts) with subtle confetti; equivalent shake + -30 pts on rejection (no shaming animation — keep it light).
  • Leaderboard podium: only screen with a colored background block. Top three get medal-tinted rank chips.
  • Streaks & badges: earned silently, shown on profile. Don't push notifications for them.

Accessibility

  • WCAG AA contrast (text ≥ 4.5:1) verified across all state badges on cream and white surfaces.
  • State changes announced via aria-live="polite"; icon labels never elided.
  • Hit targets ≥ 44 × 44 px on mobile. Reduced-motion respected — confetti disabled.

Responsibility notes

Smart container rules

  • Inject services; expose Observables to the template via async pipe — no manual subscriptions.
  • Own all dispatch: receive (@Output) from children, translate into service calls.
  • Never render UI directly — compose from presentational components only.

Presentational rules

  • ChangeDetectionStrategy.OnPush. Inputs are immutable.
  • No service injection (except TranslocoService-style infra).
  • Style files colocated; documented in Storybook with every state variant.

04

Firebase data model

Five top-level collections, with bounties and members nested under groups. Leaderboards are denormalized into a per-group subdocument and updated transactionally on approval/rejection — that keeps reads to a single document and avoids fan-out aggregation on every render.

Collection tree

users/{userId}                          // public profile, points totals
groups/{groupId}                        // group metadata + invite code
  ├── members/{userId}                 // per-group role, joinedAt, points-in-group
  ├── bounties/{bountyId}              // state machine doc
  │   └── activity/{eventId}           // timeline log
  └── leaderboard/summary             // single doc, denormalized rankings
notifications/{userId}/inbox/{id}      // per-user inbox docs
ious/{iouId}                            // pairwise debts, both-party confirm

Approval workflow (Cloud Function, transactional)

  1. Client calls approveBounty(bountyId) — callable function, OP-only enforced server-side.
  2. Transaction reads bounty + claimant member doc + leaderboard summary.
  3. Writes: bounty.state = successful + resolvedAt; claimant.points += price, .wins += 1; new IOU doc; leaderboard entries re-sorted; activity event appended.
  4. Trigger creates inbox docs for both parties; reject path mirrors with failed, .losses += 1, .points -= price, no IOU.

Security rules (sketch)

match /groups/{gid} {
  allow read:  if isMember(gid);
  allow create: if request.auth.uid == request.resource.data.ownerId;
  allow update: if isOwner(gid) || isAdmin(gid);

  match /bounties/{bid} {
    allow read:   if isMember(gid);
    allow create: if isMember(gid)
                     && request.resource.data.posterId == request.auth.uid
                     && request.resource.data.state == 'available';
    // state transitions are CF-only — clients cannot write `state` directly
    allow update: if false;
    allow delete: if resource.data.posterId == request.auth.uid
                     && resource.data.state == 'available';
  }

  match /leaderboard/summary {
    allow read:   if isMember(gid);
    allow write:  if false;   // CF-only
  }
}

function isMember(gid) {
  return exists(/databases/$(database)/documents/groups/$(gid)/members/$(request.auth.uid));
}

Index strategy

  • Composite: bounties on (state asc, createdAt desc) for the feed; (posterId, state) for the OP review queue.
  • TTL: on bounties.expiresAt + scheduled Cloud Function that flips unclaimed bounties to expired nightly.
  • Avoid: client-side aggregation across all bounties for the leaderboard — read the summary doc instead.
